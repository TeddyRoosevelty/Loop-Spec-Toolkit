# Ideation Record Template

The artifact ideation writes when the user chooses to save: a durable record of an ideation run, written as plainly as the ideas were presented in conversation.

## Template

```markdown
---
date: YYYY-MM-DD
topic: <kebab-case-topic>
focus: <optional focus hint>
---

# Ideation: <Title>

## Grounding Context

[The run's grounding summary — subject shape, notable patterns, constraints, pain
points, leverage points.]

## Focus Areas

[The areas used, one per line — or the skip reason. Omit the section if no areas
were used.]

## Ideas

### 1. <Idea Title>

[A plain description of the idea and why it matters. Explain why it holds in plain
terms, woven into the prose. Name the area it came from when that helps.]

### 2. <Idea Title>

[…one block per chosen idea…]

## Rejection Summary

| # | Idea | Reason Rejected |
|---|------|-----------------|
| 1 | <Idea> | <Reason> |
```

## Guidelines

- Keep the frontmatter exact — a later run uses it to name and find the file. Treat the body as a guide, not a form: carry a field when it adds something, drop it when it doesn't.
- Write for the person reading it later — prose that reads like the conversational presentation, not filled-in slots. Don't pad an idea with empty fields to fill the shape.
- Hold each idea in its final form. Leave out the process behind it — how it was refined, what a check changed.
- When resuming an existing record, update it in place.

## Mandatory Requirements
- Present ideas in a neutral posture. These are ideas for consideration, not well vetted solutions.
- Document only the final version of the idea, not the transitions, revisions, or intermediary steps.
- Write concisely and clearly. Prioritize the readers understanding of the idea. Avoid jargon or overly complex language; but domain language is ok and often necessary.
